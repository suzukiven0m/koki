# Spec 5: Input Validation and Error Display

**Status: READY FOR IMPLEMENTATION**

## Problem Statement

**Line 41** is the primary defect:
```rust
let Ticket { topic, endpoints } = Ticket::from_str(ticket)?;
```

When `Ticket::from_str` fails, the `?` operator propagates the raw `anyhow::Error` chain to `main()` (line 31), which returns it to the runtime. The default `anyhow` display renders the innermost cause, producing output like:

```
Error: Invalid character at position 5
```

or

```
Error: expected value at line 1 column 1
```

These are low-level error messages from `data-encoding` (line 187) and `serde_json` (line 168) respectively. The user gets no indication that the *ticket* was the problem, no hint about what a valid ticket looks like, and no clean exit code.

**Secondary issues:**

- **Line 41 only.** The `Open` branch (line 35-39) has no user input to validate. The `--name` field (line 72) is accepted as-is with no length or content check -- this is a minor concern but worth noting for completeness.
- **Line 27.** The `ticket` argument in `Command::Join` is a bare `String` with no `clap` validation. Clap parses it but does nothing to help the user.
- **`Ticket::from_str` (lines 184-190).** Returns `anyhow::Error`, which erases the structured nature of the failure. Callers cannot distinguish "bad encoding" from "bad JSON" from "good JSON but missing fields."

## Proposed Solution

Introduce a dedicated `TicketError` enum that classifies all failure modes of ticket parsing, implements `std::fmt::Display` with user-friendly messages, and is used at the call site to produce a clean `eprintln!` + `std::process::exit(1)` rather than a raw error chain dump.

**No new crate dependencies.** `data-encoding` and `serde_json` error types are already in scope. The solution uses only `std` and `anyhow`.

## Implementation Plan

**Step 1: Define `TicketError` enum (insert after line 158, before line 160)**

```rust
#[derive(Debug)]
enum TicketError {
    /// Base32 decoding failed -- the string contains non-base32 characters
    /// or has invalid length/padding.
    InvalidEncoding(data_encoding::DecodeError),
    /// Base32 decoded successfully but the bytes are not valid JSON,
    /// or the JSON does not match the Ticket schema.
    InvalidPayload(serde_json::Error),
    /// The ticket decoded and deserialized but the endpoints list is empty.
    /// This is a valid Ticket structurally but useless for joining.
    EmptyEndpoints,
}
```

**Step 2: Implement `Display` for `TicketError`**

```rust
impl fmt::Display for TicketError {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        match self {
            TicketError::InvalidEncoding(e) => {
                write!(
                    f,
                    "invalid ticket: not valid base32 encoding ({e})\n\n\
                     A ticket is a base32-encoded string. \
                     Check that you copied the full ticket and did not truncate it."
                )
            }
            TicketError::InvalidPayload(e) => {
                write!(
                    f,
                    "invalid ticket: decoded bytes are not a valid ticket ({e})\n\n\
                     This can happen if the ticket was generated by a different \
                     version of iroh-message or was corrupted."
                )
            }
            TicketError::EmptyEndpoints => {
                write!(
                    f,
                    "invalid ticket: contains no endpoint addresses\n\n\
                     A valid ticket must include at least one peer endpoint to connect to."
                )
            }
        }
    }
}

impl std::error::Error for TicketError {
    fn source(&self) -> Option<&(dyn std::error::Error + 'static)> {
        match self {
            TicketError::InvalidEncoding(e) => Some(e),
            TicketError::InvalidPayload(e) => Some(e),
            TicketError::EmptyEndpoints => None,
        }
    }
}
```

**Step 3: Update `Ticket::from_str` to return `TicketError` (replace lines 184-190)**

```rust
impl FromStr for Ticket {
    type Err = TicketError;
    fn from_str(s: &str) -> std::result::Result<Self, Self::Err> {
        let bytes = data_encoding::BASE32_NOPAD
            .decode(s.to_ascii_uppercase().as_bytes())
            .map_err(TicketError::InvalidEncoding)?;
        let ticket: Ticket = serde_json::from_slice(&bytes)
            .map_err(TicketError::InvalidPayload)?;
        if ticket.endpoints.is_empty() {
            return Err(TicketError::EmptyEndpoints);
        }
        Ok(ticket)
    }
}
```

Key change: `from_bytes` is no longer called. The two-stage decode is inlined so each stage can map to its own `TicketError` variant. The `Ticket::from_bytes` method (lines 166-169) is left in place since it may be used elsewhere or in future code.

**Step 4: Update the call site in `main()` (replace lines 40-44)**

```rust
Command::Join { ticket } => {
    let Ticket { topic, endpoints } = match Ticket::from_str(ticket) {
        Ok(ticket) => ticket,
        Err(e) => {
            eprintln!("Error: {e}");
            std::process::exit(1);
        }
    };
    println!("> joining chat room for topic {topic}");
    (topic, endpoints)
}
```

This replaces the `?` with an explicit match. The `eprintln!` goes to stderr. The exit code is 1 (general error). The `anyhow::Result` return type of `main` is unchanged -- the early exit bypasses it.

**Step 5: Validate `--name` input (optional, lightweight)**

Inside the `if let Some(name) = args.name` block (line 72), add a length check before broadcasting:

```rust
if let Some(name) = args.name {
    let name = name.trim().to_string();
    if name.is_empty() {
        eprintln!("Warning: --name was provided but is empty, skipping name announcement");
    } else if name.len() > 64 {
        eprintln!("Warning: --name is {} chars, truncating to 64", name.len());
        let name = &name[..64]; // note: may panic on char boundary; see Edge Cases
        // broadcast truncated name
    } else {
        // broadcast as-is
    }
}
```

This is lower priority. The primary fix is ticket validation. The name check can be deferred or simplified to just trimming whitespace.

## Testing Strategy

**Unit tests (add as `#[cfg(test)] mod tests` at the bottom of main.rs):**

```rust
#[cfg(test)]
mod tests {
    use super::*;
    use std::str::FromStr;

    #[test]
    fn ticket_valid_roundtrip() {
        let ticket = Ticket {
            topic: TopicId::from_bytes([0u8; 32]),
            endpoints: vec![], // will fail EmptyEndpoints check
        };
        // Need at least one endpoint for a valid ticket
        // This test validates the encoding path works
        let encoded = format!("{ticket}");
        // encoded is a valid base32 string -- but from_str will reject
        // because endpoints is empty. This is by design.
        assert!(Ticket::from_str(&encoded).is_err());
    }

    #[test]
    fn ticket_invalid_base32() {
        // Contains '8' and '9' which are not in the base32 alphabet
        let result = Ticket::from_str("not-a-valid-ticket!!!");
        assert!(result.is_err());
        let err = result.unwrap_err();
        let msg = format!("{err}");
        assert!(msg.contains("not valid base32"), "got: {msg}");
    }

    #[test]
    fn ticket_valid_base32_but_garbage_json() {
        // "AA" is valid base32 (decodes to [0x00]), but not valid JSON
        let result = Ticket::from_str("aa");
        assert!(result.is_err());
        let err = result.unwrap_err();
        let msg = format!("{err}");
        assert!(msg.contains("not a valid ticket"), "got: {msg}");
    }

    #[test]
    fn ticket_empty_string() {
        let result = Ticket::from_str("");
        assert!(result.is_err());
        // Empty string decodes to empty bytes -> JSON error
    }

    #[test]
    fn ticket_valid_json_wrong_schema() {
        // Encode a JSON object that is valid JSON but not a Ticket
        let payload = serde_json::to_vec(&serde_json::json!({"foo": "bar"})).unwrap();
        let encoded = data_encoding::BASE32_NOPAD.encode(&payload);
        let mut s = encoded;
        s.make_ascii_lowercase();
        let result = Ticket::from_str(&s);
        assert!(result.is_err());
        let msg = format!("{}", result.unwrap_err());
        assert!(msg.contains("not a valid ticket"), "got: {msg}");
    }
}
```

**Manual test steps:**

1. `cargo run -- join ""` -- expect: "invalid ticket: not valid base32 encoding" on stderr, exit 1.
2. `cargo run -- join "aa"` -- expect: "invalid ticket: decoded bytes are not a valid ticket" on stderr, exit 1.
3. `cargo run -- join "AAAA"` -- expect: base32 decodes to `[0x00, 0x00]`, JSON parse fails, same message as above.
4. `cargo run -- open` then copy the ticket, `cargo run -- join <ticket>` -- expect: success, no error.
5. `cargo run -- open`, then truncate the ticket by 5 characters, `cargo run -- join <truncated>` -- expect: base32 or JSON error with helpful message.
6. Verify all error output goes to stderr: `cargo run -- join "bad" 2>/dev/null` should print nothing to stdout.

## Risks and Edge Cases

1. **UTF-8 char boundary panic in name truncation.** The `&name[..64]` slice in Step 5 can panic if a multi-byte UTF-8 char straddles position 64. Mitigation: use `name.char_indices().nth(64).map(|(i, _)| &name[..i]).unwrap_or(&name)` or just skip truncation and let iroh handle oversized payloads. Recommendation: skip the name length check entirely for this spec; it is orthogonal.

2. **`TicketError` vs `anyhow::Error` type mismatch.** Changing `Ticket::from_str::Err` from `anyhow::Error` to `TicketError` is a breaking change for any code that calls `Ticket::from_str` and expects `anyhow::Error`. In this codebase, the only call site is line 41, and we are changing it to use `match` instead of `?`, so this is safe. The `Ticket::from_bytes` method still returns `anyhow::Error` and is unchanged.

3. **Empty ticket string.** An empty string `""` decodes via base32 to an empty byte slice `[]`, which serde_json rejects as "expected value at line 1 column 1". This falls through to `InvalidPayload`, which is correct. The message "decoded bytes are not a valid ticket" is accurate.

4. **Whitespace in ticket.** Users may paste a ticket with a trailing newline or leading/trailing spaces. The current code does not trim. `Ticket::from_str` receives whatever clap gives it. Recommendation: trim the input in the `Join` match arm before parsing: `ticket.trim()`.

5. **Exit code semantics.** Using `std::process::exit(1)` bypasses destructors and drop handlers. In this CLI tool there is no cleanup needed at the point of ticket validation (no network resources have been allocated yet -- the endpoint is bound on line 47, after ticket parsing). This is safe.

6. **Mixed error paths.** Lines 47, 69, 77, 91, 95 also use `?` and will produce raw anyhow output. This spec focuses on ticket validation only. Those paths involve network/protocol errors where the anyhow chain is more appropriate (the user needs technical details for debugging). A follow-up spec could add structured error display for those paths.

## Acceptance Criteria

1. `cargo run -- join ""` prints "invalid ticket" with a human-readable explanation to stderr and exits with code 1.
2. `cargo run -- join "not-base32!!!"` prints "invalid ticket: not valid base32 encoding" to stderr and exits with code 1.
3. `cargo run -- join <valid-base32-garbage-json>` prints "invalid ticket: decoded bytes are not a valid ticket" to stderr and exits with code 1.
4. All error messages include the original underlying error as context (e.g., the serde_json or data-encoding message) after the user-friendly prefix.
5. `cargo run -- join <valid-ticket>` continues to work exactly as before -- no regression.
6. `cargo run -- open` continues to work exactly as before -- no regression.
7. `cargo test` passes with the new unit tests covering each `TicketError` variant.
8. Error output goes to stderr, not stdout (verified by piping stdout to /dev/null).
9. No new crate dependencies are added to `Cargo.toml`.
